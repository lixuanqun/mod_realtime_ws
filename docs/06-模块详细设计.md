# 06 · 模块详细设计

## 1. mod_realtime_ws（FreeSWITCH C 模块）

### 1.1 模块构成

```
mod_realtime_ws/
├── mod_realtime_ws.c        # 模块入口、APP/API 注册、生命周期
├── rtws_session.c/.h        # 每路会话上下文与状态镜像
├── rtws_mediabug.c/.h       # media bug 读写回调(上行采集/下行写入)
├── rtws_ws_client.c/.h      # 内部协议 WS 客户端(基于 libwebsockets)
├── rtws_audio.c/.h          # 重采样、编解码、jitter buffer、帧封装
├── rtws_protocol.c/.h       # 内部协议帧编解码(二进制头 + JSON)
├── rtws_events.c/.h         # FreeSWITCH event 总线上报 & channel var
└── conf/realtime_ws.conf.xml
```

### 1.3 media bug 设计

- 通过 `switch_core_media_bug_add(session, "rtws", ..., callback, user_data, 0, SMBF_READ_STREAM | SMBF_WRITE_STREAM, &bug)`。
- 上行：`SWITCH_ABC_TYPE_READ`/`READ_REPLACE` 回调读取 L16 帧 → resample(→16k) → 封二进制帧 → WS 发送。
- 下行：从下行 jitter buffer 取帧 → `WRITE_REPLACE` 写回通道，或用 `switch_core_session_write_frame`。
- 打断：收到 `clear_playback` 即清空下行 jitter buffer，立即静音。

### 1.4 线程模型

- WS 客户端独立线程（libwebsockets service loop），与 media bug 回调线程通过无锁环形缓冲交换音频，避免阻塞媒体线程。
- 每路会话：1 个上行队列 + 1 个下行 jitter buffer，均有界。

### 1.5 配置示例 `realtime_ws.conf.xml`

```xml
<configuration name="realtime_ws.conf" description="Realtime WS">
  <settings>
    <param name="router-url" value="ws://router.internal:8080/realtime/v1/stream"/>

    <param name="default-provider" value="glm"/>
    <param name="uplink-ptime" value="20"/>
    <param name="uplink-sample-rate" value="16000"/>
    <param name="jitter-buffer-ms" value="120"/>
    <param name="jitter-buffer-max-ms" value="400"/>
    <param name="local-vad" value="true"/>
    <param name="local-vad-threshold-dbfs" value="-45"/>
    <param name="reconnect-max" value="5"/>
  </settings>

```

### 1.6 mod 保持"轻"的原则

- 不内置任何厂商 SDK/密钥。
- 只做：采集、重采样、封帧、收发、播放、本地 VAD 预暂停、事件上报。
- 所有"智能"在 router。便于 FreeSWITCH 侧零业务变更地切换厂商与策略。

## 2. realtime-router（独立服务）

### 2.1 内部分层

```
realtime-router/
├── cmd/router/main.go
├── internal/
│   ├── gateway/         # 内部协议 WS Server (mod 接入)
│   ├── session/         # 会话编排、状态机、上下文
│   ├── audio/           # 重采样、编解码、VAD、缓冲、节流
│   ├── provider/        # Provider 接口 + 各厂商 adapter
│   │   ├── glm/
│   │   ├── aliyun/
│   │   └── doubao/
│   ├── router/          # 路由策略(租户/号码/AB/负载/降级)
│   ├── tools/           # Function Call 编排(webhook 回调)
│   ├── auth/            # 厂商鉴权、token 刷新、密钥(KMS/Vault)
│   ├── config/          # profile/画像、热加载
│   └── observ/          # metrics/trace/log
└── configs/
```

### 2.2 会话编排器（Session Orchestrator）

- 每路会话一个 goroutine（或 actor），持有：内部 WS 连接、Provider 实例、音频管线、状态机。
- 输入：来自 mod 的音频帧 + 控制事件。
- 输出：归一化后的下行音频帧 + 控制事件给 mod；上行翻译后的厂商协议给 Provider。
- 通过 channel 解耦 mod 侧与 provider 侧，背压可控。

### 2.3 Provider Adapter（见 02 文档接口）

- 每厂商实现 `Provider` 接口；集中维护事件映射常量表（02 §6）。
- 豆包 adapter 内含二进制封帧/gzip；阿里云 adapter 内含 ASR+LLM+TTS 组合编排。
- adapter 仅依赖统一内部音频格式（PCM 16k 单声道），采样率转换在 audio 层完成。

### 2.4 路由策略

| 维度 | 示例 |
| --- | --- |
| 租户/号码 | 按被叫号码或租户配置默认厂商 |
| AB 实验 | 按比例灰度不同厂商/模型/音色 |
| 成本/负载 | 高峰切低价厂商，配额耗尽切备用 |
| 降级 | 主厂商故障自动切备用（05 文档） |

策略配置热加载，决策结果写 Redis 便于同租户一致性与限流统计。

### 2.5 鉴权与密钥

| 厂商 | 鉴权 | 刷新 |
| --- | --- | --- |
| GLM | API Key 签发 JWT（exp 短） | 进程内缓存，过期前刷新 |
| 阿里云 | Token（由 AccessKey 获取，有有效期）+ appkey | 定时刷新 token 池 |
| 豆包 | 火山签名(AK/SK) 或 Token | 按需 |

密钥来自 Vault/KMS，内存解密，不落盘不落日志。

### 2.7 可观测性

**关键指标（Prometheus）：**

| 指标 | 类型 | 说明 |
| --- | --- | --- |
| `rtws_active_sessions{provider}` | gauge | 活跃会话数 |
| `rtws_session_setup_ms` | histogram | 建链到 ready 时延 |
| `rtws_first_audio_ms{provider}` | histogram | 首包音频时延 |
| `rtws_barge_in_latency_ms` | histogram | 打断时延 |
| `rtws_vendor_ws_reconnect_total{provider}` | counter | 重连次数 |
| `rtws_uplink_dropped_frames_total` | counter | 上行丢帧 |
| `rtws_downlink_jitter_ms` | histogram | 下行抖动 |
| `rtws_vendor_errors_total{provider,code}` | counter | 厂商错误 |
| `rtws_tool_calls_total{name}` | counter | 工具调用 |

- **Trace**：每路会话一条 trace，span 覆盖 建链/每轮/工具调用/下行播放，贯穿 call_id。
- **Log**：结构化（call_id, session_id, provider, event），raw 厂商报文按采样落地便于排障；密钥脱敏。

## 3. 容量与性能模型

单 router 实例（参考目标，需压测校准）：

| 资源 | 估算 |
| --- | --- |
| 每会话内存 | ~1–3 MB（缓冲+上下文） |
| 每会话 CPU | 主要在重采样/(可选)Opus/VAD；PCM 直传时极低 |
| 网络/会话 | 上行 16k×16bit≈256kbps，下行 24k≈384kbps（PCM）；用 Opus 可降至 ~24–32kbps |
| 目标并发 | ≥500/实例，水平扩展 |

优化点：PCM 直传内网（省 CPU）、跨机房用 Opus（省带宽）、重采样查表/SIMD、零拷贝封帧、连接复用（豆包 FinishSession 复用连接）。

## 4. 安全

- 内部协议 token 鉴权 + 内网隔离（mTLS 可选）。
- 厂商出网 IP 白名单，统一经 router。
- 多租户隔离：密钥、配额、数据分租户；会话上下文不跨租户共享。
- 音频/转写存储加密、最小化、按期清理；满足等保/隐私合规。

## 7. 风险与未决问题（待评审确认）

1. **阿里云端到端形态**：用户给的是 ISI(ASR) 链接。是否改用阿里云"全双工/Omni 实时对话"接口直连？若坚持 ISI，则需 router 编排 ASR+LLM+TTS（时延更高）。**需确认**。
2. **豆包 event 编号**：以官方最新文档为准，需在实现期核对常量表。
3. **打断快路径**：mod 本地 VAD 是否启用（误触发 vs 时延）。倾向默认开启+消抖。
4. **内部传输**：WS vs gRPC，是否需要多路复用。倾向 WS 1:1 起步。
5. **重采样位置**：mod 侧 vs router 侧。倾向 mod 侧。
6. **上下文与降级**：跨厂商降级时上下文重建策略（摘要注入）可接受度。
7. **录音合规**：是否默认开启双轨录音及保留策略。
8. **视频**：GLM 支持视频，首期是否完全不做（倾向不做）。
