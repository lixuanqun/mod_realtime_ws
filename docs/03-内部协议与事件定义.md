# 03 · 内部协议与事件定义（mod_realtime_ws ↔ realtime-router）

本文定义 mod 与 router 之间的内部通信协议。该协议是私有的、版本化的，目标是简单、低开销、便于 C 端实现。

## 1. 传输层

- 传输：**WebSocket**（`ws://` 内网，或 `wss://` 跨域）。
- 一条 WebSocket 连接 = 一路通话会话（1:1）。简单、隔离性好。
  - 备选：多路复用（一条连接多会话，帧头带 `session_id`）。首期采用 1:1，后续若连接数成瓶颈再演进到多路复用。
- 帧类型约定：
  - **二进制帧（Binary）= 音频数据帧**（带极简头）。
  - **文本帧（Text）= 控制/事件 JSON**。
- 字节序：音频帧头中的多字节整数使用**大端序**。

### 1.1 连接建立

mod 作为客户端发起连接，URL/Header 携带会话引导参数：

```
GET /realtime/v1/stream HTTP/1.1
Upgrade: websocket
X-RT-Call-Id: <freeswitch uuid>
X-RT-Provider: glm            # glm | aliyun | doubao
X-RT-Profile: default         # router 侧预置的会话画像名
X-RT-Tenant: <租户ID>
X-RT-Token: <内部鉴权 token>
```

router 校验后接受连接，随后双方进入 §3 的握手。

## 2. 音频数据帧（二进制）

为降低 C 端解析复杂度，音频帧采用 12 字节定长头 + 裸 PCM：

```
+--------+--------+--------+--------+--------+--------+--------+--------+
| 0xR T  | ver(1) | flags(1)| codec(1)| seq (4, BE)                    |
+--------+--------+--------+--------+--------+--------+--------+--------+
| timestamp_ms (4, BE)              | sample_rate/100 (2, BE) | rsvd(2) |
+-----------------------------------+-------------------------+--------+
| PCM / 编码后音频负载 ...                                              |
+----------------------------------------------------------------------+

magic   : 2 字节 "RT" (0x52 0x54) 标识
ver     : 协议版本，当前 1
flags   : bit0 = direction(0=uplink上行, 1=downlink下行)
          bit1 = last(该 utterance/response 的最后一帧)
          bit2 = marker(标记帧，无音频，用于对齐打断点)
codec   : 0=PCM_L16, 1=OPUS, 2=G711U(PCMU), 3=G711A(PCMA)
seq     : 帧序号，单调递增（每方向独立）
timestamp_ms : 采集/生成相对时间戳
sample_rate/100 : 采样率÷100（如 16000→160, 24000→240, 8000→80）
payload : 音频字节
```

> 设计理由：定长头便于 C 端零拷贝解析；direction 位允许同一帧格式双向复用；`marker`/`last` 位用于精准打断与轮次边界。默认 codec=PCM_L16，内部链路简单可靠。

### 2.1 帧率与分片

- 上行：建议 **20ms/帧**（与 FreeSWITCH ptime 对齐），router 侧再聚合为厂商要求的 100ms 帧（GLM ≤50QPS）。
- 下行：router 按 20ms/帧切片下发，mod 端 jitter buffer 平滑播放。

## 3. 控制/事件帧（JSON 文本帧）

统一信封：

```json
{
  "v": 1,
  "id": "evt_8f3...",            // 事件唯一ID（发送方生成）
  "ts": 1751955406660,          // 毫秒时间戳
  "call_id": "<fs-uuid>",       // 通话标识
  "dir": "c2r",                 // c2r=mod→router, r2c=router→mod
  "type": "session.start",      // 事件类型，见下
  "data": { }                   // 事件负载
}
```

### 3.1 mod → router 事件（c2r）

| type | 说明 | data 关键字段 |
| --- | --- | --- |
| `session.start` | 开始一路会话，附通话/媒体信息与会话画像覆盖 | provider, profile, codec, sample_rate, caller, callee, vars{} |
| `session.stop` | 结束会话（挂机/转接） | reason |
| `audio.eof` | 上行音频结束（client_vad 手动提交场景） | — |
| `dtmf` | 用户按键，可触发业务动作 | digit, duration |
| `text.input` | 注入文本（如 IVR 转人设、初始问候触发） | text |
| `tool.result` | 业务侧 Function Call 执行结果回填 | call_id, output |
| `barge_in.detected` | mod 本地 VAD 检测到用户开口（可选，作打断快路径） | energy, ts |
| `ping` | 保活 | — |

### 3.2 router → mod 事件（r2c）

| type | 说明 | data 关键字段 |
| --- | --- | --- |
| `session.ready` | 会话与厂商均就绪，可开始送音频 | provider, model, output_sample_rate, voice |
| `session.error` | 会话级错误 | code, message, recoverable |
| `clear_playback` | **打断指令：立即清空 mod 下行播放缓冲** | reason("barge_in"/"cancel"), upto_seq(可选) |
| `response.start` | 一轮 AI 响应开始（下行音频即将到来） | response_id |
| `response.end` | 一轮 AI 响应结束 | response_id, reason |
| `user.transcript` | 用户语音转写（中间/最终） | text, is_final |
| `assistant.transcript` | AI 语音字幕（中间/最终） | text, is_final |
| `speech.started` | 检测到用户开始说话 | — |
| `speech.stopped` | 检测到用户停止说话 | — |
| `tool.call` | 模型请求调用工具 | call_id, name, arguments |
| `metrics` | 周期性质量指标 | rtt, jitter, dropped |
| `pong` | 保活响应 | — |

> 音频本身走二进制帧；这些 JSON 事件只承载控制与文本，避免把音频塞进 JSON（GLM 对外才用 base64，内部链路不必）。

### 3.3 打断（barge-in）的双路径设计

打断时延是体验核心，采用"快路径 + 准路径"：

- **快路径（可选，最低时延）**：mod 本地能量/VAD 检测到用户开口 → 立即本地清空下行缓冲并发 `barge_in.detected` 给 router。优点：~50ms 内静音；缺点：可能误触发（背景噪声）。
- **准路径（权威）**：router 依据厂商 `speech_started`/`SentenceBegin`/`ASRInfo` 或 router 自带 VAD 判定为真实打断 → 发 `clear_playback` 给 mod，并向厂商发 `response.cancel`/截断请求。

两者结合：快路径先压低 AI 音量/暂停，准路径确认后才真正取消厂商响应；若准路径在 N ms 内未确认则恢复播放（消抖）。该策略在 [05 文档](./05-会话状态机与错误处理.md) 细化。

## 4. session.start 负载示例

```json
{
  "v": 1, "id": "evt_001", "ts": 1751955406660,
  "call_id": "f2a1c9e0-...-uuid", "dir": "c2r", "type": "session.start",
  "data": {
    "provider": "glm",
    "profile": "kefu_zh",
    "codec": "PCM_L16",
    "sample_rate": 16000,
    "caller": "+8613800000000",
    "callee": "10086",
    "vars": {
      "voice": "tongtong",
      "instructions_ref": "kefu_prompt_v3",
      "vad": "server_vad",
      "greeting": "您好，这里是智能客服，请问有什么可以帮您？",
      "tools": ["query_order", "transfer_human"]
    }
  }
}
```

router 用 `profile` 加载预置会话画像（系统指令、音色、工具 schema、厂商参数），`vars` 做按呼叫覆盖，再翻译为各厂商的 `session.update`/`StartSession`/`StartTranscription`。

## 5. session.ready 与 clear_playback 示例

```json
{ "v":1,"id":"evt_101","ts":1751955407001,"call_id":"f2a1...","dir":"r2c",
  "type":"session.ready",
  "data":{ "provider":"glm","model":"glm-realtime","output_sample_rate":24000,"voice":"tongtong" } }
```

```json
{ "v":1,"id":"evt_220","ts":1751955412345,"call_id":"f2a1...","dir":"r2c",
  "type":"clear_playback",
  "data":{ "reason":"barge_in" } }
```

## 6. 版本与兼容

- 信封含 `v`，音频帧头含 `ver`。新增字段向后兼容；破坏性变更升版本并在握手协商。
- 握手阶段 mod 与 router 交换 `supported_versions`，取最高公共版本。

## 7. FreeSWITCH 事件总线映射（可选对外）

mod 可把关键内部事件转发到 FreeSWITCH 事件总线（CUSTOM 子类 `realtime_ws::*`），供 ESL/IVR 消费：

| FS Event 子类 | 触发 |
| --- | --- |
| `realtime_ws::session_ready` | session.ready |
| `realtime_ws::user_transcript` | user.transcript(is_final) |
| `realtime_ws::assistant_transcript` | assistant.transcript(is_final) |
| `realtime_ws::tool_call` | tool.call |
| `realtime_ws::barge_in` | clear_playback |
| `realtime_ws::error` | session.error |

并把最终转写写入 channel variable（如 `rt_last_user_text`/`rt_last_ai_text`），便于拨号计划与 CDR 联动。
